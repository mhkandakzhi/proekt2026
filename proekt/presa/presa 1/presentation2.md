# Цели и задачи работы

## Цель лабораторной работы

Освоение базовых сетевых протоколов (TCP/IP, HTTP/HTTPS, DNS, TLS), механизмов cookie, принципов работы луковой сети TOR и технологии беспроводных сетей Wi-Fi, а также методов обеспечения безопасности на этих уровнях.

# Процесс выполнения лабораторной работы

## Протокол HTTPS работает на прикладном уровне модели OSI/TCP/IP. UDP и TCP относятся к транспортному уровню, IP — к сетевому. HTTPS обеспечивает защищённую передачу данных между браузером и веб-сервером.

![1](image/1.png){ #fig:001 width=70% height=70% }

## TCP (Transmission Control Protocol) обеспечивает надёжную, установленную соединением доставку сегментов данных между процессами. Это ключевая функция транспортного уровня.

![2](image/2.png){ #fig:002 width=70% height=70% }

## IPv4-адрес состоит из четырёх десятичных чисел (октетов), каждое от 0 до 255. 421 > 255, 256 > 255, поэтому первые два адреса некорректны. Оставшиеся два полностью соответствуют диапазону.

![3](image/3.png){ #fig:003 width=70% height=70% }

## DNS (Domain Name System) преобразует доменные имена (например, rudn.ru) в IP-адреса, понятные маршрутизаторам. Остальные функции относятся к другим уровням.

![4](image/4.png){ #fig:004 width=70% height=70% }

## Модель TCP/IP (уровни сверху вниз): прикладной (HTTP, DNS), транспортный (TCP, UDP), сетевой (IP), канальный (Ethernet, Wi-Fi). Это стандартное представление.

![5](image/5.png){ #fig:005 width=70% height=70% }

## HTTP (HyperText Transfer Protocol) не шифрует трафик. Все данные, включая пароли и cookie, передаются в открытом виде. Шифрование добавляется протоколом HTTPS (HTTP + TLS/SSL).

![6](image/6.png){ #fig:006 width=70% height=70% }

## В HTTPS сначала выполняется TLS Handshake (согласование шифров, аутентификация сервера, обмен ключами), затем начинается фаза защищённой передачи данных. Аутентификация клиента не обязательна.

![7](image/7.png){ #fig:006 width=70% height=70% }

## В ходе handshake клиент предлагает список поддерживаемых версий TLS, сервер выбирает наивысшую общую. Это обеспечивает совместимость.

![8](image/8.png){ #fig:007 width=70% height=70% }

## Рукопожатие устанавливает параметры и генерирует ключи, но сами пользовательские данные не передаются и не шифруются на этом этапе. Шифрование начинается после завершения handshake.

![9](image/9.png){ #fig:007 width=70% height=70% }

## Куки предназначены для хранения идентификаторов сессии, настроек, user ID. IP-адрес хранится на сервере, пароль в куки не сохраняют по соображениям безопасности.

![10](image/10.png){ #fig:007 width=70% height=70% }

## Куки помогают аутентификации, персонализации, отслеживанию и статистике, но никак не влияют на надёжность сетевого соединения (это задача протоколов транспортного уровня, например TCP).

![11](image/11.png){ #fig:008 width=70% height=70% }

## Сервер в HTTP-ответе отправляет заголовок `Set-Cookie`. Браузер (клиент) сохраняет их и возвращает при последующих запросах.

![12](image/12.png){ #fig:009 width=70% height=70% }
 
## Сессионные куки не имеют срока истечения и существуют только в памяти браузера до его закрытия. После закрытия браузера они уничтожаются.

![13](image/13.png){ #fig:009 width=70% height=70% }

## Стандартная цепочка TOR состоит из трёх узлов: охранный (guard), промежуточный (middle), выходной (exit). Всего три промежуточных узла между отправителем и получателем.

![14](image/14.png){ #fig:009 width=70% height=70% }

## Отправитель знает IP получателя, чтобы инициировать соединение. Выходной узел получает IP из последнего слоя шифрования и передаёт пакет. Охранный и промежуточный узлы видят только предыдущий и следующий узел, но не конечного получателя.

![15](image/15.png){ #fig:009 width=70% height=70% }

## Отправитель устанавливает отдельные асимметричные ключи с каждым узлом цепи, создавая «луковую» структуру шифрования. Каждый узел может расшифровать только свой слой.

![16](image/16.png){ #fig:009 width=70% height=70% }

## Получатель может быть обычным веб-сервером или клиентом в Интернете. TOR работает только на стороне отправителя и узлов; трафик приходит к получателю в незашифрованном виде (если не используется дополнительное шифрование).

![17](image/17.png){ #fig:009 width=70% height=70% }

## Wi-Fi — это торговая марка, обозначающая семейство стандартов IEEE 802.11 для беспроводных локальных сетей.

![18](image/18.png){ #fig:009 width=70% height=70% }

## Wi-Fi определяет управление доступом к среде (MAC-адреса) и физическую передачу сигнала, что соответствует канальному уровню (и физическому).

![19](image/19.png){ #fig:009 width=70% height=70% }

## WEP (Wired Equivalent Privacy) имеет серьёзные уязвимости (слабая потоковая шифрация RC4, маленький вектор инициализации), легко взламывается за минуты.

![20](image/20.png){ #fig:009 width=70% height=70% }

## В современных протоколах (WPA2/WPA3) шифрование включается только после успешной аутентификации (4-way handshake). Без аутентификации данные не шифруются.

![21](image/21.png){ #fig:009 width=70% height=70% }

## WPA2 Personal (PSK) использует общий пароль, подходит для дома. WPA2 Enterprise требует сервера RADIUS и применяется в организациях.

![22](image/22.png){ #fig:009 width=70% height=70% }
